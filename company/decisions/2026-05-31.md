# Decisions - 2026-05-31

## Morning XGuard research meeting

1. 今日の事業最優先はXGuardにする。note/Today Board/その他webサービス改善は、指定パス差分確認、実Supabase migration test、real OAuth configured mode、Developer Console原価確認が進むまで後回しにする。
2. XGuard v0は、引き続き「BAN復活」ではなく「本人が事前同意したXデータのread-only backup」と「BAN後の手動再起動を支えるproof page」に限定する。
3. v0初期OAuth scopeは `tweet.read`, `users.read`, `offline.access` に維持する。`follows.read`, `dm.read`, `dm.write`, `tweet.write`, `follows.write` は初期同意画面に入れない。
4. 指定パス `/Users/uryuatsuya/XGuard/xguard` に未コミット変更があるため、昼runでは最初に差分を読む。無関係な変更は巻き戻さず、今日の実装範囲に必要な差分だけ扱う。
5. `record_api_usage_event_with_monthly_limit` は、実Supabase/Postgres上でも `service_role` 専用実行、`authenticated` 拒否、所有関係検証、同一Xアカウント整合性、月次上限超過、負値拒否を確認するまで完了扱いにしない。
6. X API原価はDeveloper Console実画面確認を正とする。確認できるまで、通常read単価と月次原価上限を保守的に使い、`Owned Reads` を第三者ユーザー向けSaaSの主前提にしない。
7. proof pageはraw X API payloadを公開せず、公開DTO、visibility、revocation、redaction、X Content削除/非公開/withheld追従を実装境界に入れる。
8. 自動DM、自動follow/unfollow、自動投稿、bulk outreach、BAN回避に見える新アカウント自動作成・誘導はv0から外す。
9. 昼の最初のゲートは `/Users/uryuatsuya/XGuard/xguard` のwrite/fetch/remote最新確認と未コミット差分確認にする。朝run時点では `exists=yes`, `writable=no`, `main...origin/main`, HEAD `d30fc48`。
10. 指定パスが書き込み不可なら、Vaultへ実装コードを置かず `/private/tmp/xguard-midday-2026-05-31` をremote最新から作って作業する。

## Midday XGuard implementation

11. OAuth configured mode確認は、`GET /api/x/oauth/status` で設定状態だけを返す方針にする。返却は `mode`, `callbackUrl`, `scopes`, `clientIdConfigured`, `clientSecretConfigured`, `writesEnabled`, `missingEnv` に限定し、`X_CLIENT_SECRET` 値、token material、`clientId` 値、`authorizationUrl` は返さない。
12. v0 OAuth scopeは昼実装後も `tweet.read`, `users.read`, `offline.access` のみに固定する。write/follow/DM scopeや `follows.read` は、Developer Console原価・policy確認が終わるまで追加しない。
13. 指定パスが書き込み不可の間は、実装コードをMyLife Vaultへ置かず、一時cloneで検証済みcommitを作り、DNS/権限復旧後に正本repoへ反映する。
