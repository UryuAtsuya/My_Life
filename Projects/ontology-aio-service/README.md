# Ontology AIO Service

Ontology と Knowledge Graph を使って、企業・商品・専門性を LLM / AI 検索に理解されやすくする SEO / AIO 支援サービス案。

## 位置づけ

- 種別: 新規事業アイデア
- 領域: SEO, AIO, GEO, LLM 検索最適化, 構造化データ
- 想定顧客: B2B SaaS、士業、クリニック、EC、専門サービス事業者、SEO 支援会社
- 状態: アイデア初期。診断 SaaS 単体ではなく、診断を入口にしたコンサル寄りサービスとして検証する

## 仮説

従来の SEO は検索エンジンにページを理解させる施策が中心だった。AI Overview や ChatGPT / Gemini / Perplexity のような LLM 検索では、ページ単位の最適化だけでなく、企業・商品・カテゴリ・競合・実績・専門用語の関係性を明確にすることが重要になる。

Ontology を使って情報の意味構造を整理すれば、LLM が回答を作るときに企業や商品の位置づけを誤解しにくくなる。サービスとしては、LLM そのものを最適化するのではなく、LLM に理解・引用・推薦されやすい情報構造を企業側に作る支援として設計する。

## サービス案

### 1. Entity / Ontology 診断

自社サイト、競合サイト、公開情報をもとに、企業・商品・カテゴリ・競合・専門用語の関係性を可視化する。

出力例:

- Entity Map
- 競合比較軸
- 自社が認識されていないカテゴリ
- LLM が混同しやすい表現
- 不足している証拠、実績、FAQ、比較情報

### 2. LLM Answer Simulation

ユーザーが実際に聞きそうな質問を投げ、AI 検索上で自社がどのように扱われるかを検証する。

質問例:

- 「〇〇に強いサービスは？」
- 「A社とB社の違いは？」
- 「中小企業向けの〇〇ツールのおすすめは？」
- 「〇〇を導入するときの注意点は？」

評価軸:

- 自社が回答に出るか
- 競合と比べてどう説明されるか
- 引用されるページはあるか
- 誤認や情報不足があるか

### 3. Content Graph 設計

記事を量産するのではなく、LLM が理解しやすい情報の接続関係を設計する。

候補コンテンツ:

- 公式カテゴリページ
- 比較ページ
- 用語集
- FAQ
- 導入事例
- 料金・プラン説明
- 競合との差分
- 専門家監修ページ

### 4. Structured Data 実装支援

schema.org などの構造化データを使い、サイト内の意味情報を明示する。

候補:

- Organization
- Product
- Service
- FAQPage
- Article
- Review
- BreadcrumbList
- Person

### 5. AIO Readiness Score

AI 検索時代にどれだけ理解されやすい状態かをスコア化する。

評価項目:

- Entity 明確性
- カテゴリ適合度
- 競合比較の明確さ
- 証拠・実績の充足度
- 構造化データの整備度
- FAQ / 用語 / 比較コンテンツの網羅度
- 外部言及・レビュー・権威性

## MVP

最初の MVP は「URL を入力すると、LLM にどう認識されているかを診断し、Entity Map / Content Gap / Schema 改善案を出すツール」にする。

最小機能:

- サイト URL の入力
- 主要ページのクロール
- 企業・商品・カテゴリ・競合候補の抽出
- LLM 質問シミュレーション
- Content Gap の抽出
- schema.org 改善案の生成
- AIO Readiness Score の表示

## 差別化

- SEO 記事作成ツールではなく、AI 検索時代のブランド理解インフラとして打ち出す
- キーワードではなく Entity / Ontology / Knowledge Graph を中心にする
- 記事案だけでなく、サイト構造・FAQ・比較軸・構造化データまで提案する
- SEO 会社向けの診断 SaaS / レポート生成ツールとしても展開できる

## リスク

- AI Overview や LLM の回答露出は外部要因が大きく、保証できない
- Ontology だけでは不十分で、被リンク、外部言及、レビュー、ブランド認知、権威性も必要
- 各 LLM / 検索サービスの仕様変化が速い
- 「AIO 対策」の市場定義がまだ曖昧
